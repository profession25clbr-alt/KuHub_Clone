Vista Lógica — Componentes del Sistema KuHub

Diagrama 1 de 4

El diagrama de Vista Lógica representa la arquitectura de componentes del sistema KuHub siguiendo el modelo de 4+1 Vistas de Kruchten. Se identifican cuatro capas funcionales interconectadas que estructuran el flujo de información desde el usuario final hasta la base de datos, reflejando una separación clara de responsabilidades y una organización orientada a módulos de negocio.

📱 Capa de Cliente

El usuario final accede al sistema a través de un navegador web convencional (Chrome, Safari o Firefox), estableciendo una conexión segura mediante HTTPS hacia el dominio appkuhub.questweb.cl. Esta capa constituye el punto de entrada al sistema y no contiene lógica de negocio propia; su función es exclusivamente presentar la interfaz e iniciar las peticiones hacia el frontend.

🖥 Capa de Presentación — Frontend (React 18 + TypeScript 5 + Vite 6)

La interfaz de usuario está compuesta por 18 pantallas cargadas de forma diferida (lazy-loading), organizadas en módulos funcionales como dashboard, inventario, solicitudes y gestión de pedidos. Los componentes visuales se construyen con la biblioteca HeroUI sobre Tailwind CSS 4. La comunicación con el backend se realiza a través de 27 clientes HTTP tipados por dominio, implementados con Axios. El estado global de la aplicación es gestionado mediante la Context API de React, con contextos específicos para autenticación, permisos dinámicos y preferencias de tema visual (AuthContext, PermissionContext y ThemeContext).

🔧 Capa de Lógica de Negocio — Backend (Spring Boot 3.4.3 + Java 21 LTS)

Las peticiones entrantes son interceptadas por una capa de seguridad que implementa autenticación mediante tokens JWT (JJWT 0.12.6), control de acceso basado en roles (RBAC) gestionado por el servicio DynamicPermissionService, y limitación de tasa de peticiones. El núcleo del sistema agrupa nueve módulos de negocio: Gestión Académica, Gestión de Inventario, Gestión de Pedidos, Gestión de Solicitudes, Gestión de Usuarios, Gestión de Proveedores, Pedido Semana Bodega, Dashboard con KPIs y analítica, y Gestión del Sistema. La infraestructura transversal incorpora un manejador global de excepciones, logging centralizado y documentación automática de la API REST mediante OpenAPI 2.8.9 (SwaggerConfig).

🐘 Capa de Persistencia — PostgreSQL 16.13

Los datos del sistema se almacenan en una base de datos PostgreSQL denominada kuhub_devs, compuesta por 38 tablas distribuidas según los módulos de negocio. Se utiliza la extensión pgcrypto para el cifrado de datos sensibles, y se aplica particionamiento semestral para optimizar el rendimiento de las consultas históricas. La comunicación con el backend se realiza mediante JDBC/JPA a través del puerto TCP 5432, y la instancia reside en una subred privada accesible vía VPC Peering en la dirección 172.26.12.228.
